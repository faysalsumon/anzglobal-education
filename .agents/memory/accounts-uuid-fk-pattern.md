# Accounts table UUID/text type mismatch

## The rule
`accounts.id` is stored as `uuid` in the actual DB despite Drizzle schema declaring `text("id")`. Any new column that FKs to `accounts.id` **must** use `uuid` type in the DB (not `text`), or PostgreSQL will reject the FK constraint with "cannot be implemented".

**How to apply:**
- When adding a new `account_id` FK column via Drizzle migration SQL, use raw SQL or `IF NOT EXISTS` guards with explicit `uuid` type.
- The migration file generated by `db:generate` will use `text` — this must be manually corrected to `uuid` before running.
- Columns already working with this pattern: `crm_contacts.sub_agent_account_id` (uuid in DB, text in schema) — these FKs were created before the mismatch was noticed.

**Why:**
PostgreSQL's FK implementation requires matching column types. The original `accounts.id` was created as `uuid` (likely via an early migration using a different ORM config) but the Drizzle schema was later written as `text`. The mismatch persists.

## Drizzle migration tracking

The `__drizzle_migrations` table is in the `drizzle` schema (not `public`). To query or insert:

```sql
SELECT * FROM drizzle.__drizzle_migrations ORDER BY created_at DESC;
```

To mark a partially-applied migration as complete (prevents re-run on next startup):
```js
const hash = crypto.createHash('sha256').update(fs.readFileSync('migrations/XXXX.sql', 'utf8')).digest('hex');
await pool.query('INSERT INTO drizzle.__drizzle_migrations (hash, created_at) VALUES ($1, $2)', [hash, timestamp]);
```

The hash is SHA-256 of the migration file's raw content. The timestamp is the `when` value from `migrations/meta/_journal.json`.

## queryClient array key URL building

`buildUrlFromQueryKey` in `client/src/lib/queryClient.ts` joins all string parts of a query key array with `/`. So `["/api/admin/accounts", id, "notes"]` becomes `/api/admin/accounts/${id}/notes`. This is correct for hierarchical cache invalidation.
